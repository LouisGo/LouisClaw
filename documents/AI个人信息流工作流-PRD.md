# AI 个人信息流工作流 PRD

> 版本：v1.0  
> 状态：可交付初稿  
> 目标运行时：优先 OpenClaw，兼容同类 Claw/Agent Runtime

---

## 1. 项目概述

本项目旨在构建一套面向个人使用的 AI 信息流工作流，将用户在 macOS 与 Android 上产生的零散信息输入统一接入，自动完成筛选、结构化处理、聚合摘要与定时推送，降低信息堆积与回顾成本。

该系统的核心目标不是“收集全部信息”，而是将高频、碎片、跨平台的信息输入压缩为少量高价值输出，形成可持续使用的个人信息处理闭环。

---

## 2. 产品目标

### 2.1 核心目标

1. 建立跨端、低摩擦的信息统一入口。
2. 自动过滤低价值噪音，提高信息信噪比。
3. 将保留内容转为结构化条目，支持归档与检索。
4. 以日报 / 周报形式输出可快速阅读的摘要。
5. 在保证体验的前提下，将模型调用成本控制在低水平。

### 2.2 成功标准

- 用户单次投递耗时不超过 10 秒。
- 系统可稳定生成日报，且日报长度可快速读完。
- 摘要内容相较原始输入显著降噪。
- 月度模型成本可控，不出现失控增长。
- 用户愿意持续使用统一入口并阅读输出结果。

---

## 3. 用户与使用场景

### 3.1 目标用户

- 当前阶段仅面向单一用户本人。
- 用户具备多设备、多平台信息输入习惯。
- 用户不愿投入大量精力做手动整理，但希望保留高价值信息。

### 3.2 典型场景

1. 在 Mac 上复制文本、代码、网页链接，希望稍后自动整理。
2. 在 Android 上从 B 站、抖音、知乎、豆瓣、微博等 App 分享内容，希望统一进入处理流。
3. 每日固定时间收到当天高价值信息摘要，而不是面对原始信息堆积。
4. 对推送结果进行简单反馈，帮助系统逐步贴合个人偏好。

---

## 4. 范围定义

### 4.1 本期范围

支持以下输入类型：

- 文本片段
- 链接
- 图片截图
- 代码片段
- 视频链接（仅处理标题、简介、链接元信息）

支持以下核心能力：

- 统一入口接收
- 原始内容暂存
- 去重与规则过滤
- AI 轻量分类与价值判断
- 定时聚合生成日报
- 单一渠道推送
- 基础归档

### 4.2 暂不纳入本期

- 自动抓取微信私域内容
- 视频 / 音频全文转录
- 知识图谱与复杂关联网络
- 多人协作与对外共享
- 重度产品化 UI

---

## 5. 产品原则

- **低摩擦输入**：用户只做投递，不做整理。
- **先过滤后理解**：优先用规则降噪，减少模型调用。
- **分级处理**：轻任务用轻模型，重任务只用于聚合总结。
- **输出优先**：优先保证“看得完、看得懂、值得看”。
- **渐进式建设**：先跑通闭环，再扩展能力。
- **运行时解耦**：工作流设计独立于具体平台实现。

---

## 6. 角色与职责

| 角色 | 职责 | 不负责 |
|---|---|---|
| 用户 | 投递内容、阅读摘要、给出反馈 | 手动分类、手动整理 |
| Claw 运行时 | 调度、触发、工具调用、流程编排 | 内容语义理解 |
| AI 模型 | 分类、去噪、摘要、聚合、提炼行动项 | 存储与调度 |
| 平台 / 工具 | 提供输入输出接口 | 统一工作流逻辑 |
| 存储层 | 保存原始条目、结构化条目与简报 | 内容判断 |

---

## 7. 总体流程

```text
用户投递
→ 统一入口接收
→ 原始队列暂存
→ 规则预处理（去重 / 过滤 / 类型识别）
→ AI 轻处理（分类 / 价值判断 / 标签 / 一句话摘要）
→ 结构化入库
→ 定时深处理（聚合 / 归纳 / 摘要）
→ 生成日报 / 周报
→ 推送给用户
→ 用户反馈
→ 调整后续筛选策略
```

## 8. 功能需求

## 8.1 统一输入

### 目标

建立一个跨端、低摩擦、可扩展的标准入口，承接所有原始信息。

### 要求

- 入口应支持 macOS 与 Android。
- 输入动作应尽可能复用系统已有能力，如复制、分享、转发。
- 入口应保留原始内容与基础上下文信息。
- 入口应作为唯一标准接入点，避免多套并行链路。

### 推荐实现

- 统一 Bot / 收件箱作为标准入口。
- Telegram 作为第一实现方案，但设计上不与 Telegram 强绑定。

---

## 8.2 原始队列与标准信息单元

### 目标

定义稳定的数据结构，保证后续处理可复用、可扩展、可追踪。

### 标准信息单元 `Item`

```yaml
id: 唯一标识
source: 来源平台或入口
capture_time: 捕获时间
device: 来源设备
content_type: text | link | image | code | mixed
raw_content: 原始内容
normalized_content: 规范化文本
url: 原始链接（可空）
title: 提取标题（可空）
tags: 标签数组
topic: 主题分类
status: pending | dropped | archived | summarized
value_score: 0-100
decision: drop | archive | digest | follow_up
reason: 决策原因
summary: 一句话摘要
dedupe_key: 去重键
```

### 要求

- 原始内容与 AI 处理结果分开存储。
- 每条记录必须可追踪状态流转。
- 每次 AI 决策必须保留理由，用于后续纠偏。

---

## 8.3 规则预处理

### 目标

在低成本前提下清除明显噪音，减少无效模型调用。

### 处理内容

- 重复检测与合并
- 极短、无语义内容过滤
- 超长内容截断与规范化
- 内容类型初步识别

### 基础规则示例

- 纯表情、纯语气词、无语义短句直接丢弃。
- 长度过短且无链接、无标题、无上下文内容降权处理。
- 相同链接或高相似文本只保留主条目。

### 输出状态

- `drop`
- `pending`
- `merged`

---

## 8.4 轻量 AI 处理

### 目标

以低成本完成基础理解与价值分层。

### 输入

- 单条或小批量 `Item`

### 输出内容

- 内容类型确认
- 主题标签
- 一句话摘要
- 价值评分
- 决策结果
- 决策原因

### 决策结果定义

- `drop`：噪音、无价值、无需保留
- `archive`：有参考价值，但不进入日报
- `digest`：应进入日报 / 周报
- `follow_up`：需要单独关注、行动或深入研究

### 价值判断标准

进入 `archive` 以上至少满足以下之一：

- 与当前重点主题相关
- 含有可复用知识或方法
- 可转化为行动
- 未来大概率需要回看
- 可与已有条目形成有效关联

---

## 8.5 定时深处理

### 目标

将分散条目聚合为用户可快速阅读的高价值摘要。

### 触发方式

- 每日一次生成日报
- 每周一次生成周报

### 处理内容

- 聚合当期 `digest` 与 `follow_up` 条目
- 按主题分组
- 合并重复观点
- 提炼核心结论
- 输出可行动项

### 输出格式

Markdown 简报，建议包含：

1. 今日 / 本周最值得关注的内容
2. 按主题分组的摘要
3. 可执行事项
4. 值得长期归档的内容

---

## 8.6 推送输出

### 目标

在固定时点向用户输出高密度、短时可读的摘要结果。

### 要求

- 默认输出为日报。
- 推送内容必须可快速读完。
- 每条摘要需说明“为什么值得看”。
- 高优先级 `follow_up` 可支持即时提醒。

### 渠道策略

- 第一优先：消息型推送渠道
- 第二优先：邮件
- 第三优先：知识库落档

---

## 8.7 归档与生命周期

### 目标

平衡可回溯性与存储清洁度，避免系统长期膨胀。

### 生命周期策略

| 数据层 | 内容 | 策略 |
|---|---|---|
| 原始队列 | 未处理原始内容 | 短期保留 |
| 结构化条目 | 标签、摘要、决策结果 | 中长期保留 |
| 日报 / 周报 | 聚合输出 | 长期保留 |
| 低价值条目 | `drop` 内容 | 定期清理 |

### 要求

- 推送完成后，条目状态需更新。
- 重复条目只保留一个主条目与引用关系。
- 原始内容保留周期应可配置。

---

## 8.8 用户反馈闭环

### 目标

让系统逐步学习用户偏好，减少长期偏差。

### 最小反馈指令

- 这条有用
- 这条没用
- 少推这类
- 多关注这类

### 反馈用途

- 调整主题权重
- 调整保留 / 丢弃阈值
- 调整摘要长度与推送频率
- 优化后续筛选规则

---

## 9. 非功能需求

### 9.1 成本

- 优先规则过滤，减少模型调用。
- 轻量任务使用低成本模型。
- 深度处理按批次执行，避免逐条调用。
- 高成本模型仅用于最终聚合与少量高价值条目。

### 9.2 性能

- 新内容进入队列后应尽快完成轻处理。
- 日报生成时间应稳定，避免长时间阻塞。

### 9.3 安全

- 最小化工具权限。
- 不暴露不必要的执行能力到公网。
- 敏感信息需支持脱敏或最小保留。

### 9.4 可迁移性

- 核心工作流不依赖特定平台语法。
- 存储格式优先使用 Markdown / 结构化文本。
- 输入、处理、输出层应尽量解耦。

---

## 10. 技术策略

### 10.1 运行策略

- 以 OpenClaw 作为第一实现载体。
- 设计保持对其他 Claw / Agent Runtime 的迁移能力。

### 10.2 模型策略

- 轻处理：低成本模型负责分类、打标、初筛。
- 深处理：更强模型负责日报 / 周报聚合与高价值总结。
- 不使用最高成本模型进行常规批处理。

### 10.3 存储策略

- 优先本地文件或轻量结构化存储。
- 外部知识库作为增强能力，而非强依赖。

---

## 11. 里程碑

### Phase 1：MVP

目标：跑通“输入—筛选—日报—推送”闭环。

- 建立统一入口
- 接收文本、链接、图片、代码片段
- 实现原始队列与 `Item` 结构
- 实现规则过滤与轻量 AI 分类
- 每日生成 Markdown 日报
- 推送到单一渠道

### Phase 2：增强

目标：提升提取质量与归档能力。

- 链接正文提取
- 知识库归档
- 重复合并与主题聚类
- 用户反馈机制上线

### Phase 3：精细化

目标：提升长期价值密度。

- 周报 / 月报
- 偏好学习
- 按主题订阅推送
- 更细粒度行动项提取

---

## 12. 验收标准

### MVP 验收

- 用户可从至少两个终端完成投递。
- 系统可稳定接收并记录原始条目。
- 系统可自动完成去重、基础筛选与分类。
- 系统可每日生成一份结构化摘要。
- 推送结果具备可读性，噪音显著低于原始输入。
- 成本处于可接受范围内。

---

## 13. 风险与应对

| 风险 | 说明 | 应对 |
|---|---|---|
| 输入链路过多 | 初期复杂度过高 | MVP 只保留一个统一入口 |
| 噪音过高 | 影响摘要质量 | 先规则过滤，再 AI 分类 |
| 平台限制 | 某些 App 难以自动化接入 | 优先手动分享，不强求全自动 |
| 成本失控 | 重模型调用过多 | 分级模型、批处理、限频 |
| 输出过长 | 用户不愿阅读 | 强制控制日报长度 |
| 偏好误判 | 摘要长期跑偏 | 引入最小反馈闭环 |
| 平台绑定 | 后续迁移困难 | 设计层与实现层解耦 |

---

## 14. 产品定位

这不是一个“收集全部信息”的系统，而是一个将跨平台碎片输入压缩为少量高价值输出的个人信息流处理系统。

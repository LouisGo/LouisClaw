# LouisClaw vNext 能力边界与分层契约

> 目的：明确 `LouisClaw` 代码仓库、`OpenClaw`、`SiYuan`、`IM` 各自负责什么，不负责什么，避免后续把高弹性语义能力过度写死到代码里，或把稳定底层能力错误上提到运行时。

## 1. 当前判断

基于当前仓库现状，这套系统已经具备较稳的本地骨架：

- `LouisClaw` 已具备 intake、process、digest、export、SiYuan API export 的基本闭环。
- 当前缺的不是更多底层流水线，而是更清晰的分层边界与更像产品协议的 skills。
- vNext 不应把所有能力都继续写进代码，也不应把所有能力都交给 `OpenClaw` 临场发挥。

一句话原则：

> 代码层负责稳定原语与可重放状态，运行时负责高语义编排与生成，SiYuan 负责沉淀与查看，IM 负责输入、提醒与轻反馈。

## 2. 四方边界表

| 组件 | 应负责 | 不应负责 | 当前状态 | vNext 重点 |
|---|---|---|---|---|
| `LouisClaw` repo | 输入契约、落盘、状态流转、去重、轻处理入口、digest/export 原语、SiYuan 映射 | 聊天体验、复杂主题写作、所有高语义判断 | 已有较稳骨架 | 收敛为稳定原语层 |
| `OpenClaw` | 意图理解、skill 路由、时间任务编排、轻/重任务分流、专题生成、提醒策略 | 主存储、隐式状态、不可复现的数据写入 | 能力强，但 skill 契约偏薄 | 强化为任务编排与生成层 |
| `SiYuan` | 可见文档、多端同步、长期沉淀、人工编辑与回看 | 主处理引擎、原始混乱输入主入口、状态真相源 | API 导出已打通 | 强化为沉淀与消费层 |
| `IM` | 低摩擦输入、标题提醒、轻反馈、随手投递 | 长文知识组织、主存储、复杂归档 | 尚未正式接入 | 强化为输入与通知层 |

## 3. repo 的能力边界

`LouisClaw` 应继续保留这些能力：

- 标准 `Item` 契约与本地落盘
- `landing/inbox/items/digests/exports/state` 目录与状态演进
- 去重键、来源字段、时间窗口、映射关系
- 标准任务入口：`process_inbox`、`build_digest`、`daily_pipeline`、`export_siyuan`
- 可验证的 `SiYuan` 导出与更新语义
- 可被任何 runtime 复用的最小 CLI 原语

`LouisClaw` 不应继续深写这些能力：

- 某主题文章应该如何组织段落
- 最近 3 天内容应该如何写成一篇高质量综述
- 用户一句自然语言指令应落到哪个内容产物
- IM 场景下该回复一句话、标题还是不回复

判断标准：

- 确定性强、要求重放、要求排错 -> 留在 repo
- 高语义、高弹性、允许风格变化 -> 放到 `OpenClaw + skill`

## 4. OpenClaw 的能力边界

`OpenClaw` 应承担：

- 根据用户意图选择 `capture / triage / synthesis / publish / feedback`
- 调用 repo 原语，而不是绕过 repo 直接制造主状态
- 把小时级轻整理与按需深整理拆开
- 在生成长文、专题、回顾时做语义编排
- 决定结果只提醒、只写入 `SiYuan`，还是两者都做

`OpenClaw` 不应承担：

- 自己维护 canonical item store
- 跳过 repo 契约直接写杂乱文件
- 让每次输入都触发一次完整重处理
- 把 `SiYuan` 当数据库来推断系统状态

关键原则：

- `OpenClaw` 可以强，但必须被 skill 契约约束
- 默认优先复用 repo 任务，不先造新脚本
- 允许 `OpenClaw` 负责“怎么写”，但不负责“主状态存在哪”

## 5. SiYuan 的能力边界

`SiYuan` 适合作为：

- 高质量 Markdown 文档的可见消费面
- 专题文、阶段回顾、digest、follow-up 的沉淀地
- 多端同步查看入口
- 你继续人工编辑与组织的地方

`SiYuan` 不适合作为：

- 主 item 存储
- 原始输入的统一总线
- 运行时状态数据库
- IM 附件与临时导出文件的存放处

因此 vNext 原则是：

- `SiYuan` 是 publish target，不是 processing engine
- repo 仍然是 source of truth
- 允许把 `SiYuan` 某些指定文档作为输入源，但必须通过显式 pull 契约进入 repo

## 6. IM 的能力边界

`IM` 适合作为：

- 90% 随手投递场景的入口
- 标题级通知与轻提醒
- 最小反馈通道，如“有用/没用/多来/少来”

`IM` 不适合作为：

- 长期知识组织界面
- 复杂结构化输出的主阅读面
- 系统 canonical state 的承载层

默认产品原则：

- 大部分 IM 输入都是 `silent capture`
- 没有明确请求时，不要求立刻回复或生成摘要
- 真正的高质量内容在后续时间窗任务或按需综合任务中产生

## 7. vNext 任务分层契约

下一阶段应把能力明确拆成四类任务，而不是一条无限膨胀的大流水线。

### 7.1 Capture

目标：低摩擦接收信息，不强求即时理解。

- 典型输入：剪贴板文本、分享链接、Markdown 片段、IM 消息、图片说明
- 默认行为：静默落盘
- 典型产物：标准化 landing/inbox 记录
- 默认不做：长回复、专题生成、深度总结

### 7.2 Hourly Triage

目标：以时间窗口做轻整理，承担“截流器”角色。

- 典型输入：最近 1 小时新增条目
- 典型处理：去重、归类、轻摘要、决策、主题升温判断
- 典型产物：处理后的 `items`、简短 triage note、候选 `digest/follow_up`
- 默认不做：长文、重模型深写作

### 7.3 Deep Synthesis

目标：按主题、按项目、按时间段生成高质量内容。

- 典型输入：近 3 天、某个 topic、某类标签、用户点名主题
- 典型产物：专题文章、阶段回顾、方法卡、重点整理文
- 默认输出：优先写入 `SiYuan`
- 可选通知：只把标题或简短说明发到 `IM`

### 7.4 Publish

目标：把内容发布到合适的消费面。

- `SiYuan`：用于可见沉淀与长期回看
- `IM`：用于提醒、标题、轻通知
- repo 导出目录：用于本地附件与中间产物

## 8. 现状与目标之间的主要缺口

| 主题 | 当前能力 | 目标能力 | 缺口 |
|---|---|---|---|
| 输入 | 已有 CLI、landing、markdown pull | 支持静默录入为默认协议 | 缺少统一的 silent capture 约束 |
| 轻处理 | 已有 process/classify/digest 骨架 | 小时级截流与主题升温识别 | triage 产物与规则尚未产品化 |
| 深处理 | 主要是 daily digest | 专题文、阶段回顾、按主题综合 | 缺少正式 synthesis 契约 |
| 发布 | 已能导出 `SiYuan` | 根据产物类型决定 publish 方式 | 缺少 publish 协议与提醒规则 |
| 反馈 | 尚无正式闭环 | IM 轻反馈写回偏好 | 缺少 feedback skill 与元数据约定 |

## 9. vNext 决策

后续演进默认遵守这些决策：

1. 不把 repo 做成“大而全智能体”
2. 不把 `OpenClaw` 当作主存储
3. 不让 `SiYuan` 承担主流程状态
4. 不让 `IM` 承担复杂知识组织
5. 优先补 skill 契约，再考虑是否需要最小代码钩子

## 10. 对当前仓库的直接含义

从当前阶段进入 vNext，最值得优先推进的不是新增复杂代码，而是：

- 明确 skill 路由：`capture / triage / synthesis / publish / feedback`
- 修正现有 skill 的默认行为，尤其是 `intake` 不再默认追问后续动作
- 为高质量产物定义稳定契约，再决定是否补最小代码接口

这意味着：

> 代码层先收敛，skill 层先变强，产品层先把边界写清。

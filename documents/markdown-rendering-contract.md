# LouisClaw Markdown Rendering Contract

> 目的：保证 digest、item export、synthesis 文稿、SiYuan 导出在 Markdown 阅读器中可点击、可扫描、可回看，而不是只把信息以“纯文本堆叠”的方式输出。

## 1. 根判断

当前问题优先不是 `SiYuan` 的问题，也不是 skill 缺失问题，而是内容渲染契约不够清楚。

也就是说：

- 数据被保留下来了
- 信息也被压缩出来了
- 但 Markdown 呈现方式没有充分利用 Markdown 的可读性能力

## 2. 渲染原则

- 有来源链接时，优先用显式 Markdown 链接，而不是裸 URL
- 标题或摘要优先做成可点击文本
- 长 URL 不直接裸露在 digest 主体里，避免影响扫描体验
- item/export 文稿里保留明确的“来源链接”字段
- 结构优先使用标题、列表、引用、简洁短句，不堆长段纯文本
- 不追求“把所有 Markdown 语法都用上”，而是选择少量高收益组件稳定复用

## 3. 最小 Markdown 组件集

为了用最低成本换最高可读性，当前优先使用这 6 类组件：

- `# / ## / ###`：标题层级
- `>`：摘要判断、总览结论
- `-`：无序列表，用于 digest 与元信息
- `1.`：有明确先后顺序时使用
- `**粗体**`：标签、字段名、重点提示
- `[文本](url)`：显式可点击链接

默认暂不把表格当成通用解法。

原因：

- 表格在移动端和窄栏阅读里可读性差
- 很多 digest/回顾内容并不是“字段对字段”的结构
- 过度表格化会降低扫描速度

只有在这些场景才优先考虑表格：

- 多个对象并排比较
- 同一批条目的固定字段对照
- 周报/月报里的趋势汇总

## 4. 模型与契约的边界

不建议把 Markdown 展示完全交给大模型自由发挥。

更合适的方式是：

- 重复性强、每天都会出现的产物：用稳定契约约束
- 高语义、一次一写的长文：允许模型自由发挥，但仍遵守最小组件集

这样做的原因是：

- 模型天然会写 Markdown，但风格容易漂移
- 没有约束时，不同产物的阅读体验会越来越不一致
- 固定产物如果没有稳定结构，就很难比较、验证、持续优化

## 5. 具体约定

### Daily Digest

- digest 列表项中，如果条目有 URL：
  - 使用 `[摘要](url)`
- 不再默认输出裸 URL 作为尾巴
- `先看这个` 区域中的主条目也应尽量可点击
- `先看这个` 中的字段名优先使用粗体，提升扫描性

### Item Export

- 若存在来源链接，使用：
  - `- 来源链接: [打开原文](url)`
- 保持原文正文单独放在 `## 原文`
- 元信息字段优先用粗体标签呈现

### SiYuan Export

- 写入 `SiYuan` 的 Markdown 同样遵守显式链接规则
- 不依赖阅读器对裸 URL 的自动识别
- 不为了追求“花哨”引入依赖阅读器特性的非通用语法

## 6. 为什么这样做

这样做的目标不是“更花哨”，而是：

- 提升回看速度
- 提升点击成功率
- 降低长链接对 digest 扫描性的破坏
- 让 `SiYuan` 和其他 Markdown 阅读器中的表现更稳定

## 7. 后续延伸

后续如果继续强化 Markdown 可读性，可再考虑：

- 为 digest 条目增加统一的“来源”短标签
- 为 synthesis 文稿追加“参考条目”链接区
- 对图片、视频、网页卡片设计不同的渲染模板

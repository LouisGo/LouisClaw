# AI 个人信息流工作流 · OpenClaw 落地实施方案

> 文档目的：把 PRD 进一步下钻为可实施方案，明确如何优先基于 OpenClaw 跑通这套个人信息流工作流。  
> 适用范围：个人自用，优先服务“我自己在多设备、多平台上的信息流整理需求”。

---

## 1. 真实需求还原

这套系统首先不是一个“第二大脑产品”，也不是一个“面向大众的信息管理平台”。

它要解决的是我自己的真实问题：

1. 我在不同设备上持续产生碎片信息输入：
   - macOS 上复制文本、代码、图片、链接
   - Android 上从 B 站、抖音、知乎、豆瓣、微博、微信等平台分享或转发内容
2. 这些信息大多只短暂停留，不会被手工整理。
3. 我需要 AI 替我做初筛、去噪、摘要、聚合和定时回顾。
4. 这套流程需要能远程同步，不依赖单一设备，不要求我手动维护复杂系统。
5. 这套流程优先跑在 OpenClaw 上，但设计上应兼容同类 Claw / Agent Runtime。
6. 目标是“多快好省”：
   - 不追求全自动抓取一切
   - 不追求炫技式产品设计
   - 不接受高昂 token 成本与长期维护负担

一句话概括：

> 这是一个“个人信息流压缩系统”，目标是把跨设备、跨平台的碎片输入压缩成少量值得回看的输出。

---

## 2. 落地目标

### 2.1 核心目标

优先跑通以下闭环：

```text
多端投递 → 统一收件 → 自动初筛 → 定时聚合 → 日报推送 → 轻反馈纠偏
```

### 2.2 本期成功标准

- 我能在 Mac 和 Android 上稳定投递内容。
- 原始内容可进入统一队列，不依赖手工整理。
- 系统可自动过滤明显噪音，并保留高价值条目。
- 每天能收到一份可快速看完的摘要。
- 推送内容明显优于原始信息堆积。
- 成本可控，维护动作可接受。

---

## 3. 设计原则

- **统一入口优先**：先解决“怎么收”，再优化“怎么理解”。
- **OpenClaw 优先实现**：但不把工作流设计绑死在某个技能或某个平台上。
- **远程同步优先**：入口、队列、摘要都应可跨设备访问。
- **轻依赖优先**：尽量用 Bot、Webhook、文件存储、定时任务，少依赖重型基础设施。
- **低成本优先**：规则过滤 > 轻模型 > 重模型。
- **先能用，再变强**：MVP 只做闭环，不做完美。

---

## 4. 总体架构

```text
[输入层]
  Mac 剪贴板 / 手动发送 / 浏览器复制 / Android 分享
        ↓
[统一入口]
  Bot / Inbox / Webhook
        ↓
[OpenClaw Runtime]
  接收任务 / 调度流程 / 调用工具 / 触发 cron
        ↓
[预处理层]
  去重 / 过滤 / 标准化 / 类型识别
        ↓
[AI 轻处理层]
  价值判断 / 分类 / 标签 / 一句话摘要
        ↓
[存储层]
  原始队列 / 结构化条目 / 日报文件
        ↓
[AI 深处理层]
  每日聚合 / 主题归纳 / 行动项提炼
        ↓
[推送层]
  Telegram / 邮件 / 其他消息渠道
        ↓
[反馈层]
  有用 / 没用 / 少推 / 多推
```

## 5. OpenClaw 中的模块划分

## 5.1 入口模块

### 目标
接收来自不同设备和平台的原始内容，写入统一收件队列。

### 建议实现
- 用一个统一 Bot / Inbox 作为标准入口。
- Mac 端优先支持：复制后发送、剪贴板投递、手动转发。
- Android 端优先支持：系统分享菜单转发。
- 微信场景默认接受“手动转发”，不强行自动化。

### 输入要求
每次进入入口时尽量记录：
- 来源平台
- 来源设备
- 捕获时间
- 原始内容
- 原始链接（如有）

---

## 5.2 队列与标准化模块

### 目标
把所有输入统一转为标准 `Item` 结构，便于后续 AI 和归档模块复用。

### 标准结构

```yaml
id: 唯一标识
source: 来源入口或平台
device: 设备
capture_time: 捕获时间
content_type: text | link | image | code | mixed
raw_content: 原始内容
normalized_content: 规范化文本
url: 链接
status: pending | dropped | archived | summarized
value_score: 0-100
decision: drop | archive | digest | follow_up
tags: []
topic: 主题
summary: 一句话摘要
reason: 决策理由
dedupe_key: 去重键
```

### 实现原则
- 原始数据与处理结果分开存储。
- 任意环节都可重放，不因单次失败而丢数据。
- 每条内容必须可追踪状态变化。

---

## 5.3 预处理模块

### 目标
尽量在不调用大模型的前提下去掉明显噪音。

### 处理内容
- 去重
- 内容长度与结构检查
- 超长截断
- 链接 / 文本 / 图片 / 代码初步识别
- 简单黑名单过滤

### 建议规则
- 纯表情、纯语气词、无语义短句直接丢弃。
- 相同链接多次出现仅记录一次主条目。
- 短文本但带链接时不直接丢弃。
- 图片优先提取附带文案，OCR 仅作补充，不做默认重度处理。

---

## 5.4 轻处理模块

### 目标
用低成本模型完成初筛。

### 输入
- 单条 `Item`
- 或每批 5~20 条小批量 `Item`

### 输出
- `decision`
- `value_score`
- `tags`
- `topic`
- `summary`
- `reason`

### 决策逻辑
- `drop`：无意义、过于临时、纯噪音
- `archive`：有参考价值，但不值得打断我
- `digest`：适合进入日报
- `follow_up`：需要行动、研究或单独提醒

### 真实需求约束
该模块不是为了“分类得多漂亮”，而是为了回答一个更实际的问题：

> 这条东西以后还值不值得我再看一眼？

---

## 5.5 深处理模块

### 目标
把当日高价值条目压缩成一份我愿意看的摘要。

### 触发方式
- 每日固定时间生成日报
- 每周固定时间生成周报

### 输出内容建议
- 今日最值得看：3~5 条
- 分主题摘要
- 需要行动的事项
- 值得归档但不需立即处理的内容

### 摘要要求
- 可在 3~5 分钟内读完
- 不堆砌原文
- 尽量解释“为什么值得看”
- 保留原始链接以便回溯

---

## 5.6 存储模块

### 目标
让内容可追踪、可回看、可低成本维护。

### 推荐存储分层

| 层级 | 建议载体 | 用途 |
|---|---|---|
| 原始队列 | 本地文件 / 轻量数据库 | 保存未处理内容 |
| 结构化条目 | 本地文件 / SQLite / 轻量 KV | 保存分类、标签、决策 |
| 摘要输出 | Markdown 文件 | 保存日报 / 周报 |
| 长期归档 | 可选接入知识库 | 长期检索与沉淀 |

### 落地原则
- MVP 优先本地 Markdown + 轻量结构化存储。
- 外部知识库仅做增强能力，不作为 MVP 前提。

---

## 5.7 推送模块

### 目标
把结果送到我真正会看的地方。

### 优先级建议
1. Telegram 或同类消息渠道
2. 邮件
3. 知识库落档

### 推送要求
- 日报为主
- 高优先级事项支持即时提醒
- 推送内容不超过我愿意快速读完的长度

---

## 5.8 反馈模块

### 目标
让系统逐步贴合我的偏好，而不是永远静态地工作。

### 最小反馈指令
- 这条有用
- 这条没用
- 少推这种
- 多关注这种

### 反馈落地方式
- 写回到条目元数据
- 调整主题权重
- 调整摘要长度与推送阈值
- 更新规则白名单 / 黑名单

---

## 6. 部署方案

## 6.1 方案 A：本机优先

### 适用场景
- 当前以个人试验和低成本验证为主
- OpenClaw 长时间运行在自己的 Mac 上

### 优点
- 部署快
- 调试方便
- 成本低

### 风险
- 离线时可能中断
- 对单机稳定性有依赖

---

## 6.2 方案 B：VPS 优先

### 适用场景
- 希望 24/7 接收与定时处理
- 更重视远程同步和稳定运行

### 优点
- 长驻稳定
- 远程可访问
- 更适合作为统一队列与调度中心

### 风险
- 需要额外运维
- 安全边界需要更谨慎

---

## 6.3 推荐路径

推荐采用：

```text
MVP 阶段：本机验证闭环
稳定后：将统一入口、队列和定时任务迁移到 VPS
```

这样既符合“先跑通再增强”，也更贴合你的真实需求：

> 先验证我是不是真的会用，再为持续运行付出额外成本。

---

## 7. OpenClaw 中建议的流程编排

## 7.1 流程一：新内容接收

```text
触发：Bot / Inbox 收到消息
动作：
1. 保存原始内容
2. 生成 Item
3. 执行去重与规则过滤
4. 进入轻处理
5. 写入结构化存储
```

## 7.2 流程二：日报生成

```text
触发：每日 cron
动作：
1. 拉取当天 digest / follow_up 条目
2. 进行聚合与主题归类
3. 生成日报 Markdown
4. 保存日报
5. 推送到目标渠道
6. 更新条目状态
```

## 7.3 流程三：反馈纠偏

```text
触发：用户回复反馈指令
动作：
1. 识别反馈目标条目
2. 写入反馈标签
3. 更新主题偏好或过滤阈值
4. 记录后续策略变更
```

## 8. 模型与成本策略

### 8.1 模型分工
- 轻模型：分类、打标、初筛、一句话摘要
- 强模型：日报 / 周报聚合、少量高价值内容深度提炼

### 8.2 成本控制原则
- 优先规则过滤
- 按批处理，不逐条重模型调用
- 视频只处理标题、简介、链接
- 图片只做必要 OCR
- 周报尽量基于日报二次聚合，而不是回头重扫原始内容

### 8.3 真实需求约束
成本控制不是附加项，而是这套系统能否长期存在的前提。

如果系统需要大量 token 才能跑通，它就不适合你的个人长期使用场景。

---

## 9. 风险与边界

| 风险 | 本质问题 | 应对策略 |
|---|---|---|
| 想接所有平台 | 复杂度迅速失控 | 先统一入口，不做全平台直连 |
| 想全自动抓取 | 权限、稳定性、维护成本高 | 以分享 / 转发为主，自动化为辅 |
| 摘要不准 | 个人偏好难以一次命中 | 引入反馈闭环 |
| 推送过长 | 最终没人看 | 强制控制长度与层级 |
| 系统过重 | 维护成本高于收益 | 坚持文件化、轻依赖、少组件 |
| 过度产品化 | 偏离个人真实需求 | 始终以“我自己是否真的在用”为检验标准 |

---

## 10. 实施建议

### 10.1 先做什么
先只做四件事：
- 一个统一入口
- 一个原始队列
- 一个轻处理流程
- 一个日报推送流程

### 10.2 暂时不要做什么
MVP 阶段不要优先投入：
- 多入口并行
- 全自动网页抓取
- 知识图谱
- 复杂 UI
- 视频转录
- 面向他人的产品包装

### 10.3 真正的验证标准
不是“功能是否丰富”，而是：

1. 我是否持续愿意投递内容
2. 我是否愿意读日报
3. 日报是否真的比原始信息流更有用
4. 我是否愿意为这套系统持续保留运行成本

---

## 11. 交付建议

本方案建议拆为两层交付：

### 交付层 1：可运行 MVP
- 统一入口
- 原始队列
- 规则过滤
- AI 初筛
- 日报生成
- 单渠道推送

### 交付层 2：可优化系统
- 反馈闭环
- 主题偏好
- 周报聚合
- 知识库归档
- 远程常驻部署

---

## 12. 一句话结论

这套系统最正确的落地方式，不是从“万能信息中台”开始，而是从“我每天愿意用的一条收件入口 + 一份值得看的日报”开始。
